# 설계 우선 원칙

> 원본: `blueprints/design-principles.md`

## 1. 개발 단계 워크플로우

작업 요청을 받으면 **작업 유형에 따라 적절한 단계**를 밟는다.

| 작업 유형 | 프리셋 | 적용 단계 |
|----------|--------|----------|
| 버그픽스, 설정 변경, 문서 수정 | **Small** | 구현 → 검증 |
| 기능 추가, 리팩토링, API 변경 | **Medium** | 아이디어 구체화 → 설계 → 구현 → 검증/테스트 |
| 아키텍처 변경, 새 모듈, 대규모 마이그레이션 | **Large** | 아이디어 구체화 → 설계 → 설계검토 → 구현 → 검증 → 테스트 |

**단계 결정**: 작업 요청 시 AI가 유형을 추론하고 한 줄로 제안 → 사용자 승인/조정 후 진행.
판단이 애매하면 한 단계 위 프리셋을 제안한다.
프로젝트별 CLAUDE.md에서 분류 기준을 오버라이드할 수 있다.

## 2. 산출물 관리

각 단계의 산출물은 `docs/artifacts/` 하위 단계별 디렉토리에 저장한다:

```
docs/artifacts/
├── ideation/        ← 단계 1
├── design/          ← 단계 2
├── review/          ← 단계 3
├── summary/         ← 단계 5
└── test-report/     ← 단계 6
```

- 파일명: `<3자리 번호>.<작업명-kebab>.md` (예: `001.add-auth-module.md`)
- 같은 작업은 모든 디렉토리에서 동일 번호·동일 이름 공유
- Small은 `summary/`만, Medium은 `ideation/` + `design/` + `summary/`(+ 선택 `test-report/`)

### 코드 설명 문서

코드 구현 시 `docs/source/`에 소스 트리와 동일한 구조로 코드 설명 문서를 생성/갱신한다:
- 파일명: 소스 확장자를 `.md`로 변경 (예: `uart.c` → `docs/source/src/drivers/uart.md`)
- 내용: 모듈 목적, 주요 함수/클래스 역할, 의존 관계, 주의사항
- 코드 주석과 중복하지 않는다 — 주석은 "왜(why)", 문서는 "전체 그림(what/how)"

## 3. 서브에이전트 파이프라인

각 단계를 **서브에이전트(별도 컨텍스트)**가 실행한다.

- 서브에이전트는 **이전 단계 산출물 파일만 참조** — 대화 맥락에 의존하지 않는다
- 단계 간 **사용자 확인 게이트** — 산출물 확인 → 승인 후 다음 단계 진행
- 메인 세션은 전체 흐름 조율과 사용자 소통 담당
- 특정 단계 재실행 시 별도 세션에서 해당 산출물 경로를 지정하여 실행 가능

### 단계별 산출물 내용

| 단계 | 산출물 핵심 내용 |
|------|-----------------|
| 아이디어 구체화 | 목적, 범위(scope), 제약조건 |
| 설계 | 접근 방식, 트레이드오프, 대안, 작업 항목, 영향 범위 |
| 설계 검토 | 피드백 이력, 변경사항, 최종 승인 기록 |
| 검증 | 변경 파일, 핵심 로직, 사용 방법, 한계/개선점 |
| 테스트 | 실행 명령어, 결과, 실패 항목 분석 |

더 나은 방법이 있으면 설계 단계에서 **능동적으로 대안을 제안**한다.
