# 버전 정책

> My_AI_manual 지침 라이브러리의 버전 관리 정책을 정의한다.

---

## 1. 버전 체계

**SemVer + Local**: `Major.Minor.Patch.Local`

```
v0.1.0      ← 라이브러리 버전 (3자리)
v0.1.0.3    ← 프로젝트 버전 (4자리 — Local은 프로젝트에서만 사용)
```

- **앞 3자리** (`Major.Minor.Patch`) = 라이브러리가 관리
- **4번째** (`.Local`) = 프로젝트에서 지침을 수정할 때마다 증가

---

## 2. 버전 증가 기준

### 라이브러리 (3자리)

| 버전 | 기준 | 예시 |
|------|------|------|
| **Major** | 기존 프로젝트에 **수동 마이그레이션 필요** | 디렉토리 구조 변경, 필수 항목 추가/삭제, 템플릿 구조 변경 |
| **Minor** | 새 지침/스킬 추가 — 기존 프로젝트에 **선택적 채택** 가능 | 새 스킬, 새 blueprint, 워크플로우 단계 추가 |
| **Patch** | 의미 변경 없는 수정 | 오타, 표현 개선, 명확화, 예시 보강 |

> **v0.x.y 기간**: Major를 올리지 않고 Minor가 breaking change 역할을 겸한다.
> 충분히 안정화된 후 v1.0.0을 선언한다.

### 프로젝트 (4번째 Local)

| 상황 | 동작 |
|------|------|
| 프로젝트에서 지침 수정 | Local +1 (예: `v0.1.0.0` → `v0.1.0.1`) |
| 라이브러리 업데이트 적용 | 앞 3자리를 라이브러리 버전으로 갱신, Local을 0으로 리셋 |

---

## 3. 릴리스 시점

**묶음 릴리스** — 의미 있는 변경이 모이면 태그한다.

```
commit: feat: 워크플로우 6단계 도입
commit: feat: 서브에이전트 파이프라인
commit: docs: 표현 개선
→ 태그: v0.1.0
```

- 매 커밋마다 버전을 올리지 않는다
- 관련 변경을 묶어서 하나의 릴리스로 태그
- `VERSION` 파일과 git tag를 동시에 갱신

---

## 4. 버전 기록 위치

| 위치 | 내용 |
|------|------|
| `VERSION` (루트) | 현재 라이브러리 버전 (`0.1.0`) |
| git tag | 릴리스 태그 (`v0.1.0`) |
| `CHANGELOG.md` | 버전별 변경 이력 |
| 프로젝트 `CLAUDE.md` 헤더 | `기반: my_ai_manual v0.1.0.0` |
| `workstations/<alias>.deploy.json` | 워크스테이션별 배포 프로젝트 버전 현황 |

---

## 5. 배포 추적

### 구조

```
workstations/
├── registry.md              ← 전체 현황 인덱스
├── <alias>.json             ← 워크스테이션 환경 상태 (기존)
└── <alias>.deploy.json      ← 배포된 프로젝트 목록
```

### deploy.json 스키마

```json
{
  "workstation": "<alias>",
  "projects": [
    {
      "name": "프로젝트명",
      "path": "~/project/path",
      "base_version": "0.1.0",
      "local_version": 0,
      "deployed_date": "YYYY-MM-DD",
      "last_synced": "YYYY-MM-DD"
    }
  ]
}
```

## 6. 흡수 워크플로우

프로젝트에서 발전한 지침을 라이브러리로 역반영하는 프로세스.

```
프로젝트 (v0.1.0.3) — Local 수정 3회 누적
    ↓ 흡수 판단
라이브러리에서 프로젝트 CLAUDE.md와 기반 버전의 diff 비교
    ↓ 항목별 판단
  채택 → blueprints/ 또는 해당 파일에 반영 → Minor 또는 Patch 증가
  미채택 → 프로젝트 고유로 유지
    ↓ 라이브러리 릴리스 후
프로젝트에 새 버전 적용 → 앞 3자리 갱신, Local 리셋
```

> 향후 `/absorb` 스킬로 자동화 예정 — diff 비교, 항목별 채택 판단을 대화형으로 지원.

---

## 7. 업데이트 흐름

라이브러리 새 버전을 기존 프로젝트에 적용하는 프로세스.

1. 라이브러리에서 새 버전 릴리스 (`v0.2.0`)
2. 프로젝트에서 `/project-init` 재실행 (Case 2: 기존 프로젝트 갱신)
3. 프로젝트 CLAUDE.md의 기반 버전을 `v0.2.0.0`으로 갱신
4. `deploy.json`의 `base_version`, `last_synced` 갱신
5. `registry.md` 갱신
