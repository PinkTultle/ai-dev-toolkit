# 설계 우선 원칙

> 도구 무관 공유 지식 — 모든 AI 코딩 어시스턴트에 적용

---

## 1. 개발 단계 워크플로우

작업 요청을 받으면 **작업 유형에 따라 적절한 단계**를 밟는다.

### 전체 단계 (Large 기준)

| 단계 | 산출물 디렉토리 | 핵심 활동 |
|------|----------------|----------|
| **1. 아이디어 구체화** | `ideation/` | 목적·범위·제약조건을 사용자와 1문 1답으로 명확화 |
| **2. 설계** | `design/` | 접근 방식, 트레이드오프, 대안, 영향 범위 문서화 |
| **3. 설계 검토** | `review/` | 사용자 피드백 → 수정 → 재검토 반복, 승인 기록 |
| **4. 구현** | (코드) | 확정된 설계에 따라 구현 |
| **5. 검증** | `summary/` | 설계 의도 대비 구현 결과 확인, 기술 부채 명시 |
| **6. 테스트** | `test-report/` | 빌드·단위·통합 테스트 실행 및 결과 보고 |

### 작업 유형별 프리셋

작업 유형으로 단계를 결정한다. 프로젝트 규모가 아니라 **개별 작업의 성격**이 기준이다.

| 작업 유형 | 프리셋 | 적용 단계 |
|----------|--------|----------|
| 버그픽스, 설정 변경, 문서 수정 | **Small** | 4 → 5 |
| 기능 추가, 리팩토링, API 변경 | **Medium** | 1 → 2 → 4 → 5/6 |
| 아키텍처 변경, 새 모듈, 대규모 마이그레이션 | **Large** | 1 → 2 → 3 → 4 → 5 → 6 |

### 단계 결정 규칙

1. 작업 요청을 받으면 AI가 유형을 추론하고 **한 줄로 제안**한다
   - 예: `이 작업은 Medium으로 판단합니다 (기능 추가). 설계부터 시작할까요?`
2. 사용자가 승인하면 해당 프리셋으로 진행
3. 사용자가 조정하면 (예: "설계 검토도 넣어줘") 그에 따라 단계 추가/제거
4. 프로젝트별 CLAUDE.md에서 분류 기준을 오버라이드할 수 있다

> 판단이 애매하면 **한 단계 위 프리셋**을 제안한다 — 단계를 줄이는 건 쉽지만 빠뜨린 단계를 나중에 끼워넣기는 어렵다.

## 2. 산출물 관리

### 2.1 디렉토리 구조

```
<project>/
└── docs/
    ├── artifacts/              ← 작업 단위 산출물
    │   ├── ideation/           ← 단계 1: 아이디어 구체화
    │   ├── design/             ← 단계 2: 설계 계획서
    │   ├── review/             ← 단계 3: 설계 검토 기록
    │   ├── summary/            ← 단계 5: 구현 요약
    │   └── test-report/        ← 단계 6: 테스트 결과
    └── stack/                  ← 기술 스택별 지침 (누적)
```

`stack/`은 프로젝트에서 발견한 패턴·컨벤션을 기술 스택별로 누적하는 디렉토리다.
- 파일명: `<언어 또는 프레임워크>.md` (예: `rust.md`, `react.md`)
- 사용자와 합의된 내용만 기록 — AI가 임의 추가하지 않는다

`source/`는 소스 코드 트리를 미러링하여 코드별 설명 문서를 관리하는 디렉토리다.

```
docs/source/          ← 소스 트리와 동일한 구조
├── src/
│   ├── app/
│   │   └── main.md           ← src/app/main.c 설명
│   └── drivers/
│       └── uart.md           ← src/drivers/uart.c 설명
└── lib/
    └── protocol.md           ← lib/protocol.c 설명
```

**규칙:**
- 코드 파일(또는 모듈)을 구현·수정할 때 대응하는 문서를 `docs/source/` 아래 동일 경로에 생성/갱신한다
- 파일명: 소스 파일명에서 확장자를 `.md`로 변경 (예: `uart.c` → `uart.md`)
- 디렉토리 단위로 묶는 게 자연스러우면 디렉토리명 `.md`도 허용 (예: `drivers.md`)
- 내용: 모듈 목적, 주요 함수/클래스 역할, 의존 관계, 주의사항
- 코드 주석과 중복하지 않는다 — 주석은 "왜(why)", 문서는 "전체 그림(what/how)"

### 2.2 파일 네이밍

```
<작업번호>.<작업명>.md
```
- 작업번호: 3자리 자동 증가 (`001`, `002`, ...)  — 해당 프로젝트의 전체 디렉토리에서 기존 최대 번호 + 1
- 작업명: kebab-case 영문
- 같은 작업의 산출물은 **모든 디렉토리에서 동일 번호·동일 이름** 공유

예: `docs/artifacts/design/001.add-auth-module.md`

### 2.3 문서 헤더

모든 산출물 파일은 다음 헤더로 시작한다:

```markdown
# <작업번호>. <작업명>

- **날짜**: YYYY-MM-DD
- **작업 개요**: 한 줄 설명

---

(이하 단계별 본문)
```

### 2.4 프리셋별 산출물

해당 프리셋에 포함되지 않는 단계의 디렉토리에는 파일을 생성하지 않는다.

| 프리셋 | 생성되는 산출물 |
|--------|---------------|
| **Small** | `summary/` |
| **Medium** | `ideation/`, `design/`, `summary/` (+ 선택적 `test-report/`) |
| **Large** | 전체 5개 디렉토리 |

## 3. 단계별 실행 방식

### 3.1 서브에이전트 파이프라인

각 단계를 **서브에이전트(별도 컨텍스트)**가 실행한다.

```
[메인 세션] ─── 단계 판단 / 사용자 확인 게이트 관리
    ├── [서브에이전트 1] 아이디어 구체화 → ideation/*.md 저장
    │        ↓ (사용자 확인)
    ├── [서브에이전트 2] 설계 → ideation/*.md 읽기 → design/*.md 저장
    │        ↓ (사용자 확인)
    ├── [서브에이전트 3] 설계 검토 → design/*.md 읽기 → review/*.md 저장
    │        ↓ (사용자 확인)
    ├── [메인 or 서브에이전트] 구현 → design/*.md + review/*.md 참조
    │        ↓
    ├── [서브에이전트 5] 검증 → 코드 + design/*.md 비교 → summary/*.md 저장
    │        ↓
    └── [서브에이전트 6] 테스트 → test-report/*.md 저장
```

**핵심 규칙:**
- 서브에이전트는 **이전 단계 산출물 파일만 참조** — 대화 맥락에 의존하지 않는다
- 단계 간 **사용자 확인 게이트** — 산출물 확인 → 승인 후 다음 단계 진행
- 메인 세션은 전체 흐름 조율과 사용자 소통을 담당
- 특정 단계만 재실행하려면 **별도 세션에서 해당 산출물 경로를 지정**하여 실행 가능

### 3.2 단계별 산출물 내용

| 단계 | 산출물 핵심 내용 |
|------|-----------------|
| **아이디어 구체화** | 목적, 범위(scope), 제약조건 |
| **설계** | 접근 방식, 트레이드오프, 대안, 작업 항목, 영향 범위 |
| **설계 검토** | 피드백 이력, 변경사항, 최종 승인 기록 |
| **구현** | (코드 — 별도 산출물 없음) |
| **검증** | 변경 파일, 핵심 로직, 사용 방법, 한계/개선점 |
| **테스트** | 실행 명령어, 결과, 실패 항목 분석 |

더 나은 방법이 있으면 설계 단계에서 **능동적으로 대안을 제안**한다.

## 4. 기술 부채 명시

임시 방편이나 개선이 필요한 코드에는 반드시 표시:
```c
// TODO(이유): 설명 — 추후 [더 나은 방법]으로 개선 필요
// FIXME(긴급도): 설명 — 현재 [문제점] 발생 가능
// HACK: 설명 — [조건]에서만 동작하는 임시 코드
```
